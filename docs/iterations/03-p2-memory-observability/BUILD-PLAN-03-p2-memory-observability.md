## Dependency order & concurrency

- F-08 (multi-turn memory) and F-09 (observability) are INDEPENDENT at the contract level and can be built CONCURRENTLY. F-08 mutates conversation-thread plumbing (router input list, both boundary adapters, GeneratorState.messages, generate-node seed). F-09 adds a new observability package and wraps the SAME call sites (router, coach, generator, logger nodes) with timing context managers. They touch overlapping files (hub.py, generator/graph.py) but in non-conflicting regions (F-08: data threading; F-09: a wrapping context manager around the same ainvoke). Build them in separate worktrees and reconcile at integration.
- Within F-08 build TEST-FIRST in this order: (1) no-doubling regression test [PASSES today — locks current behavior, drives NO chat.py change]; (2) coach double-append RED test -> fix _coach_boundary_node; (3) router-context RED test -> fix _router_node input; (4) 'make it shorter' CRITICAL-PATH RED test -> add GeneratorState.messages field, thread it through generator boundary + emit workout-summary AIMessage, seed _generate_node. Then full backend suite.
- Within F-09 build TEST-FIRST: redaction + JSON-event-shape unit tests -> observability/logging.py module -> set ContextVar in chat.py boundary -> instrument router/coach/logger/generator call sites + retries -> env-gated tracer -> critical-path reconstructable+redacted integration test -> full suite (assert router dispatch seam unbroken).
- F-07 is NOT among this iteration's drafted feature plans (only F-08 and F-09 were drafted). Its substrate is already-implemented and its contracts are frozen above as already-implemented/reuse references. If F-07 is added to this iteration, it is independent of both F-08 and F-09 and can run concurrently — it touches the generator SELECTION path (filtering) and Reason emission, distinct from F-08's message threading and F-09's logging wrappers.

**Concurrency:** F-08 and F-09 run concurrently in separate worktrees. The only shared files are backend/app/graph/hub.py and backend/app/agents/generator/graph.py. Conflict surface is low and non-semantic: F-08 changes the DATA passed into ainvoke (message lists, prior context) and what boundaries return; F-09 WRAPS those same ainvoke call sites in `with llm_call(role, model):` / emits tool_call+retry events. At integration both edits coexist in the same functions — expect mechanical merge conflicts in _router_node, _coach_boundary_node, _generator_boundary_node, and generator _generate_node/_make_gate_node, resolvable by keeping F-08's message-threading inside F-09's timing wrapper. The schema-aware fake-model seam (conftest.py) is the shared invariant for BOTH: neither feature may break it. F-08 relies on the fake ignoring invocation input (RunnableLambda(lambda _: ...)) so router/dispatch tests stay green after the router input becomes a message list; F-09 relies on the monkeypatch of get_model still landing because instrumentation lives at call sites, not inside the factory. Run the FULL backend suite (uv run pytest) once after integrating both — specifically tests/graph/test_hub_dispatch.py and test_router_node.py (seam-sensitive) plus the new test_session_memory.py and test_observability_request.py.

## Iteration risks

- Integration conflict in hub.py/generator graph.py: F-08 and F-09 edit the same four functions. Mitigation: F-08's message-list threading must end up nested INSIDE F-09's `with llm_call(...)` context manager around each ainvoke; resolve by composition, not by picking one side.
- Live coach double-append bug (verified): router appends HumanMessage(user_message) to HubState.messages BEFORE the coach boundary, the boundary forwards state['messages'] (now including the current turn), and _answer_node re-appends HumanMessage(user_message) — coach model sees the current turn TWICE. Frozen fix: boundary forwards list(messages)[:-1] when last is the current HumanMessage. A naive 'just forward messages' multi-turn fix amplifies this every turn.
- Do NOT add first-invocation gating in chat.py to 'prevent accumulator doubling' — empirically there is NO doubling on the pinned langgraph (add_messages treats [] as a no-op; 2-turn probe = [H,A,H,A]). Gating risks DROPPING the current user_message on resume turns. Lock absence of doubling with a regression test that PASSES today; only if it ever goes red is gating warranted.
- Do NOT round-trip subgraph_result/WorkoutPayload through the checkpoint to recover the prior workout — hits the msgpack 'Deserializing unregistered type' warning (Route/CoachResult/Reason/WorkoutPayload), a hard failure under future langgraph/LANGGRAPH_STRICT_MSGPACK. Carry prior workout as a plain-string AIMessage summary + plain selected_exercise_ids. Keep langgraph version-bump OUT of this iteration (deferred hardening item).
- GeneratorState.messages MUST be a plain list[BaseMessage] with NO reducer (mirrors shipped coach/state.py:16). Adding add_messages to it would create a shared mutable key reduced in two places, violating ADR-004. It is input-only prior context the generate node reads, never writes back; default [] so single-turn tests are unaffected.
- F-09 must NOT proxy the BaseChatModel surface to time LLM calls: router uses .with_structured_output(...).ainvoke, generator uses .bind_tools(...).ainvoke, coach uses .ainvoke — a proxy that only wraps .ainvoke misses the structured/bound chains. Frozen approach: emit llm_call events at each CALL SITE via a context manager (role + MODEL_CONFIG[role] are available there), NOT by wrapping get_model's return.
- F-09 must NOT reassign/decorate the get_model module symbol — conftest monkeypatches app.models.factory.get_model AND app.graph.hub.get_model. Instrumentation that replaces the symbol is silently defeated by the monkeypatch (tests) and could swallow the fake (prod ordering). Keep get_model the one-line factory; instrument at call sites.
- F-09 correlation id: subgraph nodes (coach/generator) do NOT receive session_id (isolated-state contract). Use a contextvars.ContextVar set once in chat.py._stream_chat; it propagates across await boundaries within the same task. Default 'unknown' when unset (direct subgraph tests).
- F-09 redaction + vendor-tracer tests are vacuous unless made real: redaction test must monkeypatch.setenv a sentinel OPENROUTER_API_KEY and assert it never appears; vendor-tracer must be env-gated with lazy try/except ImportError and asserted to no-op false when no key present. langsmith stays a transitive dep — do NOT add to required deps.
- Assertions against the schema-aware fake are vacuous if they check the fake's OUTPUT reflects context (the fake ignores its input). F-08 must assert at the SEAM: spy/capture the message list handed to the router's structured model and to the generator's bound model, and assert prior-turn content is present — never assert the fake's returned decision/workout reflects context.
