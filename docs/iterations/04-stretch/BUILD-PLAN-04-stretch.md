## Iteration risks

- **F-11 contract symmetry:** the `ExplanationEvent` signature must be identical on backend (`streaming.py`) and frontend (`types/api.ts`) — freeze it first, then build both sides. The exhaustive `encode_sse` (`assert_never`) and `reduceSSE` (`assertNever`) switches will fail the build if a variant arm is forgotten — that is the intended safety net, not a bug.
- **F-11 summarization is load-bearing, not cosmetic:** `hub.py:260–278` emits one reason **per muscle group AND per equipment item** per included exercise — a 6-exercise workout yields 30+ triples. The `explanationLines` humanizer MUST group by `(claim, relation)` and collapse distinct objects, or the panel is noise and AC2 fails. Test the dedupe explicitly.
- **F-11 graceful degradation:** hide the panel when `explanationLines(reasons)` is empty (covers no-reasons turns AND `note`-only coach triples) — return `null`; and gate the SSE emit on `response.explanation` truthiness so coach/log turns send nothing.
- **F-10 ambiguous-case scoring:** `RoutingDecision.route` is a **required** field — never score by `decision.route == expected`. An ambiguous case's correct outcome is `decide_route` returning `(None, clarification)`; label cases by the gated outcome (`expected_route=None` / `expects_clarification=True`). A `None` decision (structured-output failure) is a normal data point, not a crash. This is the single easiest thing to get wrong in F-10 — pin it in the case-set test and the smoke test.
- **F-10 node refactor on a hot path:** delegating `_router_node` to `classify` is guarded by existing hub/router/model-swap tests; run `tests/critical/test_model_swap_routing.py` after. Approver may opt to keep `classify` harness-private (prompt is single-sourced either way).
- **F-10 stays a tiny seed:** the roadmap explicitly forbids a formal eval pipeline. No metric framework, pandas, charts, results DB, YAML loader, or argparse framework. ~120-line module + ~15-line CLI + one smoke test.
- **F-12 prompt-perturbation:** composing the preamble into coach/generator/logger and adding a sentence to the router prompt can ripple into prompt-sensitive tests (`test_router_node.py`, generator tool-loop, logger extraction). Mitigation: **ADD voice only, never reword functional directives; run the FULL backend suite, not just touched files.**
- **F-12 subjectivity (spec-acknowledged):** the objective floor is testable (copy non-empty, sourced from `app.voice`, partnership markers, ≠ the old bare generic string); overall warmth/cohesion and the card-agents' (generator/logger) voice are **human-review** against BRAND.md. Generator/logger emit cards, not prose — their AC1 contribution is fallback copy + preamble future-proofing, not a testable behaviour change today.
- **F-12 import cycle:** `voice.py` needs `ClarificationPrompt` (in `routing.py`) while `routing.py` will import from `voice.py` — resolve one-directionally or via a plain-tuple return.
- **Integration merge points (all trivial, non-semantic):** `routing.py` and `hub.py` (F-10 ∩ F-12) and `chat.py` (F-11 ∩ F-12) are edited in different functions/lines. Resolve by composition. The schema-aware fake-model seam (`conftest.py`) is the shared invariant for all backend work — no feature may break it. Run the FULL backend + frontend suites once after integrating all three.
- **Dataset caveat (inherited, not gating):** all 18 `bilateral_pair_id` values in `data/exercises.json` are dangling (STATUS.md). F-11's `added·bilateral_pair_of` "paired both sides" line therefore won't fire on live data — only against the in-memory reciprocal fixture used in tests. F-11's tests should use such a fixture for the pairing line; do not block on the dataset.
